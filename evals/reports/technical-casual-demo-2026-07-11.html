1. Cache invalidation

Baseline

Cache invalidation is the process of updating or removing stale entries from a cache when the source of truth changes. Common strategies include time-based expiration, write-through, cache-aside, and explicit invalidation. Each strategy trades consistency, latency, and implementation complexity differently.

Skill-Assisted

Imagine you run an online store. A product page is fast because its title, price, and inventory are cached. Then someone in the admin panel changes the price from $39 to $29. If the cache keeps serving the old version, your site is now fast and wrong, which is a bad combo. That tension is the whole cache invalidation problem. The easiest fix is to let cached data expire after a fixed amount of time. That works, until you notice the awkward window where the source of truth has changed but the cache has not. Now we have a real tradeoff to work through.

2. Reference counting memory management

Baseline

Reference counting associates a count with each allocated object. Every new reference increments the count and every removed reference decrements it. When the count reaches zero, the object can be reclaimed. This approach is simple to implement but does not handle cycles on its own.

Skill-Assisted

Think of each object as carrying a little sticky note with a number on it. That number tells us how many live references still point at the object. If the number is 3, three different places still care about it. If the number drops to 0, nobody is using it anymore, so we can clean it up. Start with one object named UserProfile. A variable points to it, so the count is 1. Pass that object into a cache, and the count becomes 2. Drop the original variable, and the count falls back to 1. Once the cache lets go too, the count reaches 0 and cleanup can happen.

3. Base64 encoding

Baseline

Base64 is an encoding scheme that converts binary data into a restricted text alphabet of 64 characters. It works by taking input bytes in groups of three, turning them into 24 bits, and then splitting those 24 bits into four 6-bit chunks. Each 6-bit chunk maps to one Base64 character.

Skill-Assisted

Base64 is what you reach for when you need to move raw bytes through a system that behaves better with plain text. The bytes are still the real payload. Base64 just repackages them into a safer-looking alphabet. The easiest way to picture it is this: take 3 bytes, line them up into one 24-bit strip, then cut that strip into 4 equal 6-bit pieces. Those 6-bit pieces are small enough to map cleanly into a 64-character alphabet. Once you see that repacking step, the rest of Base64 feels much less mysterious.
